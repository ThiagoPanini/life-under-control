# Radar de codebases de produção — como backends Python resolvem os eixos do mapa

Research do ticket [#224](https://github.com/ThiagoPanini/life-under-control/issues/224) do mapa wayfinder [#219](https://github.com/ThiagoPanini/life-under-control/issues/219). Pergunta: como backends Python de produção reconhecidos resolvem os eixos ports/contratos, DI, forma de use-case, DTOs/validação, modelo de erro, configuração, async, transações, testes e topologia. Objetivo é ser um radar empírico — descrever o que codebases reais fazem, com prova de arquivo/linha —, **não** decidir o que o LUC faz. Onde a prática de produção contradiz doutrina conhecida (Clean/Hexagonal/DDD, e a própria doutrina do `apps/api` no CONTEXT.md e ADRs), o conflito é **anotado, não resolvido** — a adjudicação é dos grillings e dos tickets de decisão.

## Método

Cada achado aponta pra um permalink com SHA fixado (`github.com/<org>/<repo>/blob/<sha>/<path>#L<n>`), lido diretamente do arquivo no commit — nenhum achado vem de resumo de terceiro, README ou post. As linhas foram conferidas via `gh api` no raw do arquivo no SHA fixado; os quatro codebases delegados a subagentes tiveram os achados mais carga-de-prova reconferidos por amostra contra o arquivo. A separação entre **sinal** (padrão recorrente entre codebases independentes) e **anedota** (idiossincrasia de um só) é feita por eixo. O recorte inclui frameworks diferentes de propósito (FastAPI síncrono, FastAPI assíncrono, Pyramid, Django/DRF) pra distinguir o que é padrão de "backend Python" do que é idiossincrasia de um framework.

## Espécimes e recorte

| Codebase | Papel | Framework / forma | SHA fixado |
| --- | --- | --- | --- |
| **Netflix Dispatch** (`Netflix/dispatch`) | Produção — gestão de incidentes da Netflix | FastAPI + SQLAlchemy **síncrono** | `dd2837e82a0bf5565b1b4b4b91ea30b7262d4061` |
| **Polar** (`polarsource/polar`) | Produção — plataforma de monetização/pagamentos | FastAPI + SQLAlchemy 2.0 **async** (asyncpg) | `034c7474e763201b934369756fc221ca7e9848d2` |
| **Prefect** (`PrefectHQ/prefect`, `src/prefect/server`) | Produção — orquestrador de workflows | FastAPI **async** end-to-end | `60025750fb7ea1d69fef8a1c8b1e013cdc8149ae` |
| **Warehouse** (`pypi/warehouse`) | Produção — o backend do PyPI | **Pyramid** (não FastAPI), WSGI síncrono | `f576313a510043e07f72dd4db75c4084025566da` |
| **Sentry** (`getsentry/sentry`) | Produção — observabilidade, monólito gigante | **Django/DRF**, WSGI síncrono | `c4a0d0de19ef2e0132c02ae3b5745520861b43d2` |
| **full-stack-fastapi-template** (`fastapi/full-stack-fastapi-template`) | **Template** do autor do FastAPI — baseline canônica, **não** produção | FastAPI + SQLModel, síncrono | `4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8` |

Por que estes seis. Dispatch e Polar são os dois espécimes FastAPI de produção mais limpos e opostos (síncrono/fat-service vs. async/repository+service). Prefect é um segundo FastAPI async de produção, pra confirmar se o que Polar faz é sinal ou anedota. Warehouse e Sentry entram **de propósito** não sendo FastAPI: eles testam se um padrão é "FastAPI-shaped" ou "backend-Python-shaped" — Warehouse é o exemplo mais forte de ports+DI explícitos em Python de produção (zope.interface + container), e Sentry é o teste de escala extrema (monólito de ~20k arquivos). O template entra rotulado como template do autor do framework, útil como baseline do "shape mínimo canônico" — nunca citado como produção.

Considerados e cortados: Litestar/Starlette/Databases (são frameworks, não apps); Airflow/Home-Assistant (shape de orquestração/IoT, não backend HTTP de domínio); os vários "fastapi-best-practices"/boilerplates (são opinião/template, não produção com dado real); Zulip/Read-the-Docs (Django de produção, mas redundantes com Sentry no eixo). O recorte de 6 já dá diversidade de framework suficiente pra separar sinal de idiossincrasia.

## Detalhe por eixo

### 1. Ports / contratos

**Sinal:** porto existe **onde há um segundo implementador real** (multi-backend de DB, integração externa plugável, adapter trocável por env, RPC cross-silo) — e aí é modelado com o que o ecossistema oferecer (zope.interface, `abc.ABC`, base-class duck-typed), **quase nunca** `typing.Protocol`, e **nunca** "um Protocol por use-case". Onde não há necessidade de substituição (CRUD comum, o template), não há porto.

- **Warehouse** — o caso mais forte: `interfaces.py` por contexto com `zope.interface.Interface` e o serviço concreto com `@implementer`; 17 arquivos `interfaces.py` no repo. [`IUserService` interfaces.py L52](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/interfaces.py#L52), [`@implementer(IUserService) class DatabaseUserService` services.py L79](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/services.py#L79), [dois adapters do mesmo porto: `SMTPEmailSender`/`SESEmailSender` L58/L86](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/email/services.py#L58).
- **Dispatch** — sistema de plugins: base-class que levanta `NotImplementedError` (não ABC, não Protocol), registrada por metaclasse; o comentário admite ter sido copiada do Sentry. [`IPlugin` v1.py L42](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/plugins/base/v1.py#L42), [`# stolen from getsentry/sentry` v1.py L28](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/plugins/base/v1.py#L28), [`StoragePlugin` bases/storage.py L12](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/plugins/bases/storage.py#L12).
- **Sentry** — `RpcService(abc.ABC)` com métodos `@rpc_method`+`@abstractmethod` e adapter local `DatabaseBackedUserService`; **porto só existe na fronteira cross-silo**, não em cada use-case. [`RpcService(abc.ABC)` service.py L209](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/hybridcloud/rpc/service.py#L209), [`class UserService(RpcService)` L25](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/users/services/user/service.py#L25), [`class DatabaseBackedUserService(UserService)` impl.py L56](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/users/services/user/impl.py#L56).
- **Prefect** — `abc.ABC` como estratégia de backend de DB (Postgres/SQLite) composta num `PrefectDBInterface`. [`PrefectDBInterface` interface.py L55](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/database/interface.py#L55), [`class BaseDatabaseConfiguration(ABC)` configurations.py L120](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/database/configurations.py#L120).
- **Polar** — anedota reveladora: usa `typing.Protocol` (`RepositoryProtocol`) mas **não como seam de injeção** — os serviços instanciam a classe de repositório concreta direto, o Protocol só tipa o `self` dos mixins. [`class RepositoryProtocol[M](Protocol)` base.py L35](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/kit/repository/base.py#L35), [`OrderRepository.from_session(session)` service.py L358](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/service.py#L358).
- **Template** — nenhum porto/interface; `crud.py` fala direto com o ORM. [`app/crud.py` (funções, sem contrato) L10](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/crud.py#L10).

**Anedota:** `zope.interface` (só Warehouse, é idiomático de Pyramid); `abc.ABC` para backend plugável (Prefect); base-class duck-typed + registry por metaclasse (Dispatch, copiado do Sentry). **Conflito com doutrina:** "defina um porto/Protocol pra toda dependência" não é observado; o porto é pragmático, puxado por uma segunda implementação concreta. Polar tem o Protocol mas confessadamente não o usa como fronteira trocável.

### 2. Injeção de dependência (DI)

**Sinal:** DI de dois níveis é universal nos apps: (a) a injeção request-scoped do próprio framework para sessão/auth/paginação (FastAPI `Depends`; Pyramid `find_service`), e (b) **singletons de módulo criados no import** / service-locator para serviços de domínio e adapters. Injeção por construtor de colaboradores aparece só dentro de factories (Warehouse) ou repositórios (Polar). **Nenhum** usa biblioteca de container de DI (`dependency-injector` etc.).

- **Warehouse** — o único container de verdade: `pyramid_services` como IoC; registro por `config.register_service_factory(factory, IFoo)`, resolução por `request.find_service(IFoo)`. [`config.include("pyramid_services")` config.py L647](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/config.py#L647), [`register_service_factory(database_login_factory, IUserService)` accounts/__init__.py L89](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/__init__.py#L89), [`request.find_service(IUserService, ...)` views.py L264](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/views.py#L264).
- **Polar** — grafo puro de FastAPI `Depends` (sessão, auth) + serviço como singleton de módulo chamado direto (não via `Depends`). [`session = Depends(get_db_read_session)` endpoints.py L101](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/endpoints.py#L101), [`order = OrderService()` (singleton de módulo) service.py L3056](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/service.py#L3056).
- **Dispatch** — `Depends` para o HTTP (`DbSession`, `CommonParameters`) + service-locator por string pra resolver adapters de plugin. [`DbSession = Annotated[Session, Depends(get_db)]` core.py L162](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/database/core.py#L162), [`plugin_service.get_active_instance(..., plugin_type="oncall")` service.py L51](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/incident/service.py#L51).
- **Prefect** — `Depends(provide_database_interface)` na rota + um descriptor bespoke `db_injector` na camada `models/`, ambos passando por um provider global que lê um dict de módulo (service-locator). [`db: PrefectDBInterface = Depends(provide_database_interface)` flow_runs.py L104](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/api/flow_runs.py#L104), [`db_injector.__call__` dependencies.py L263](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/database/dependencies.py#L263).
- **Sentry** — sem container: serviço vira singleton de módulo por `create_delegation()` no import; o manager de options segue o mesmo padrão; ORM é `Model.objects`. [`user_service = UserService.create_delegation()` service.py L355](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/users/services/user/service.py#L355), [`get = default_manager.get` options/__init__.py L62](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/options/__init__.py#L62).
- **Template** — só FastAPI `Depends`, encadeado (`SessionDep` → `CurrentUser`). [`SessionDep = Annotated[Session, Depends(get_db)]` deps.py L26](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/api/deps.py#L26), [`CurrentUser = Annotated[User, Depends(get_current_user)]` L49](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/api/deps.py#L49).

**Anedota:** o container IoC de verdade (Warehouse/Pyramid) é o outlier; o descriptor `db_injector` (Prefect) é bespoke. **Conflito com doutrina:** "injete tudo por construtor / por um container" não é a norma — singleton no import + `Depends` do framework dominam; construtor-injection só dentro de factories.

### 3. Forma de use-case

**Sinal:** o use-case é **procedural** — uma função (ou um saco de métodos stateless) que **recebe a sessão/uow** e opera sobre um objeto persistência-shaped. Não é um agregado de domínio rico com comportamento; não é um objeto command/handler. Empacotado como funções livres (Dispatch, Prefect, template-crud, Sentry-`usecases`) ou como classe de serviço stateless (Polar, Warehouse), a constante é: stateless, sessão-entra, sem entidade de domínio com métodos. "Serviço gordo / view gorda / manager gordo" é a norma.

- **Dispatch** — módulo de funções livres keyword-only sobre `db_session`, orquestrando cross-concept e commitando inline. [`def create(*, db_session, incident_in)` service.py L166](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/incident/service.py#L166).
- **Polar** — classe de serviço **stateless** (sem `__init__`, todo método recebe `session` como 1º arg, um singleton por módulo) com split repository+service; e **coexiste** um `ResourceService[Model, Create, Update]` genérico herdado por mais de uma dúzia de módulos menores — duas formas de use-case ao mesmo tempo. [`class OrderService:` service.py L346](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/service.py#L346), [`class ResourceService[...]` kit/services.py L54](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/kit/services.py#L54).
- **Prefect** — funções de módulo `async def` em `models/*.py` recebendo `AsyncSession` (+ `db` injetado); a lógica de transição de estado é fatorada num stack separado de "regras" async-context-manager em `orchestration/rules.py`. [`@db_injector async def create_flow_run(db, session, ...)` models/flow_runs.py L62](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/models/flow_runs.py#L62), [`orchestration/rules.py` L534](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/orchestration/rules.py#L534).
- **Warehouse** — classe de serviço que implementa a Interface, construída por uma factory de assinatura `(context, request)`; recebe a sessão (`request.db`) e colaboradores já resolvidos. [`def database_login_factory(context, request)` services.py L970](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/services.py#L970).
- **Sentry** — **sem camada formal** de use-case para CRUD: a lógica mora no método da view DRF e em `Manager` custom gordo; pacotes novos (ex. `replays`) juntam orquestração em `usecases/` mas como funções de módulo, não interactor. [`ProjectManager(BaseManager[...])` models/project.py L176](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/models/project.py#L176), [`def delete_replays(...)` replays/usecases/delete.py L70](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/replays/usecases/delete.py#L70).
- **Template** — split entre `crud.py` fino e lógica **inline na rota** (items nem passa por crud). [`def create_user(*, session, user_create)` crud.py L10](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/crud.py#L10), [`def create_item(...)` com `session.add/commit` na rota items.py L61](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/api/routes/items.py#L61).

**Anedota:** o split repository+service de Polar (o mais próximo de camadas limpas); o rule-engine separado de Prefect pra máquina de estados; a factory `(context, request)` de Warehouse; o `ResourceService` genérico de Polar (mini-framework). **Conflito com doutrina:** o "modelo de domínio rico / agregados com comportamento" do DDD tático é praticamente **ausente**; regra de negócio às vezes mora **no próprio modelo ORM** — o oposto de anemia, mas também não é camada de use-case/interactor: [`raise RefundAmountTooHigh(self)` no model Polar order.py L438](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/models/order.py#L438).

### 4. DTOs / validação

**Sinal:** schema separado de entrada vs. saída vs. (às vezes) persistência, por concept, é quase universal (variantes Create/Update/Read). O DTO de saída é construído **direto da linha do ORM** (`from_attributes` / SQLModel / `serialize()`) — ninguém mapeia entidade→DTO à mão numa camada de mapeamento. A validação acontece na borda (fronteira da request).

- **Polar** — Pydantic v2, base `Schema` com `from_attributes=True`, split C/U/R genuíno; a instância ORM cruza pra borda HTTP como DTO via `response_model` + `from_attributes`. [`class Schema(BaseModel): model_config = ConfigDict(from_attributes=True)` kit/schemas.py L25](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/kit/schemas.py#L25), [`class OrderCreate(...)` schemas.py L297](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/schemas.py#L297).
- **Prefect** — schemas partidos por módulo: `core.py` (ORM-shaped, `from_attributes`), `actions.py` (entrada, `extra="forbid"`), `responses.py` (saída), `filters.py` (query). [`ORMBaseModel` docstring "ORM representation" bases.py L216](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/utilities/schemas/bases.py#L216), [`"""Reduced schemas for accepting API actions."""` actions.py L1](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/schemas/actions.py#L1).
- **Dispatch** — Pydantic `DispatchBase` com `from_attributes=True`, **separado** do `Base` SQLAlchemy mas **co-locado** no mesmo `models.py` por concept; Create/Read/Update por concept. [`class DispatchBase(BaseModel)` models.py L113](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/models.py#L113).
- **Template** — SQLModel **unifica** ORM e Pydantic: uma base compartilhada gera a tabela e os DTOs (Create/Update/Public). [`class UserBase(SQLModel)` models.py L14](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/models.py#L14), [`class User(UserBase, table=True)` L52](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/models.py#L52).
- **Sentry** — **três** mecanismos: DRF serializer pra entrada (com lógica dentro do `.save()`), um registry `serialize()`/`@register` separado pra saída, e `RpcModel` (pydantic) pro payload RPC. [`@register` api/serializers/base.py L29](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/api/serializers/base.py#L29), [`class RpcModel(pydantic.BaseModel)` rpc/__init__.py L42](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/hybridcloud/rpc/__init__.py#L42).
- **Warehouse** — **WTForms** na borda da view (não Pydantic); forms mutáveis presos ao request, com efeito colateral dentro do `validate_<campo>`. [`class LoginForm(PasswordMixin, UsernameMixin, wtforms.Form)` forms.py L454](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/forms.py#L454).

**Anedota:** WTForms com validators que causam efeito (Warehouse, era pré-Pydantic); os três sistemas paralelos de schema do Sentry; a unificação SQLModel do template. **Conflito com doutrina:** "modelo de domínio ≠ persistência ≠ DTO, com mapeamento explícito" **não** é observado — `from_attributes`/SQLModel **acoplam** o DTO ao shape do ORM de propósito; Prefect até docstring o schema de API como "a representação ORM". A "validação como etapa pura" é violada em Warehouse (validators com side-effect, ex. `validate_password` que dispara e-mail e desabilita a senha).

### 5. Modelo de erro

**Sinal:** hierarquia de exceção própria levantada do núcleo + **tradução centralizada pra HTTP na borda do app** é o padrão dominante de produção (middleware do Dispatch, handler do Polar, dict de handlers do Prefect, catch-all do Sentry). O núcleo levanta erros semânticos; a borda mapeia.

- **Polar** — hierarquia `PolarError(Exception)` carregando `message`/`status_code`/`headers`; levantada fundo nos serviços (e até nos models); HTTP mapeado **uma vez** central. [`class PolarError(Exception)` exceptions.py L11](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/exceptions.py#L11), [`polar_exception_handler` → `JSONResponse(status_code=exc.status_code, ...)` L17](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/exception_handlers.py#L17).
- **Prefect** — duas camadas (`PrefectException` → exceções do server); mapeamento HTTP documentado na docstring do núcleo; tradução central como dict `exception_handlers` do FastAPI, inclusive registrando `sa.exc.IntegrityError` (exceção do SQLAlchemy) como chave → 409. [`ObjectNotFoundError` exceptions.py L4](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/exceptions.py#L4), [`sa.exc.IntegrityError: integrity_exception_handler` server.py L981](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/api/server.py#L981).
- **Dispatch** — erros de domínio como subclasses de **`PydanticValueError`** (com `code`/`msg_template`) — afloram como erro de validação (422) pelo `ExceptionMiddleware`. [`class NotFoundError(PydanticValueError)` exceptions.py L15](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/exceptions.py#L15), [`class ExceptionMiddleware` main.py L220](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/main.py#L220).
- **Sentry** — duas camadas: `exceptions.py` puro (sem HTTP) + `api/exceptions.py` sobre `APIException` do DRF; catch-all `handle_exception_with_details` vira 500 `{detail, errorId}`. [`class ResourceDoesNotExist(APIException): status_code = 404` api/exceptions.py L17](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/api/exceptions.py#L17), [`handle_exception_with_details` api/base.py L311](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/api/base.py#L311).
- **Warehouse** — `pyramid.httpexceptions` levantada **direto na view** + `Exception` puro por contexto (sem base comum). Idiossincrasia extrema: a API legada de upload embute a mensagem de erro na **reason phrase HTTP**. [`raise HTTPUnauthorized` views.py L250](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/views.py#L250), [`_exc_with_message` (mensagem na status line) forklift/utils.py L6](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/forklift/utils.py#L6).
- **Template** — `fastapi.HTTPException` levantada **inline** em deps e rotas, sem hierarquia de domínio. [`raise HTTPException(status_code=404, detail="Item not found")` items.py L55](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/api/routes/items.py#L55).

**Anedota:** Dispatch pendura o erro de domínio no sistema de erros do Pydantic; Warehouse levanta `HTTPException` do framework na view e usa a reason-phrase como canal de mensagem; o template não tem hierarquia. **Conflito com doutrina:** "erro de domínio agnóstico de transporte" é **frequentemente violado** — `PolarError` carrega `status_code`/`headers`, a hierarquia do Sentry herda de `APIException` (status embutido), Prefect documenta status HTTP na docstring da exceção. O split limpo (erro puro de domínio + mapeamento só na borda) só é parcialmente realizado. Nenhum dos seis usa RFC7807/problem+json — contraste direto com o `problem+json` central do `apps/api` ([#197](https://github.com/ThiagoPanini/life-under-control/issues/197)); o formato de corpo de erro é ad-hoc (`{detail: ...}`, `{error, detail}`, `{detail, errorId}`).

### 6. Configuração

**Sinal:** objeto de settings tipado carregado de env é o default moderno — mas **partido por geração**: os apps da era FastAPI (Polar, Prefect, template) usam `pydantic-settings BaseSettings`; os mais antigos / de outro framework (Dispatch `starlette.config`, Warehouse dict imperativo, Sentry django settings) **não**. "pydantic-settings" é sinal da geração FastAPI, não universal. Env-como-fonte é universal.

- **Polar** — `pydantic_settings.BaseSettings`, ~170 campos tipados, `env_prefix="polar_"`, arquivo `.env` escolhido de `POLAR_ENV`; singleton global. [`class Settings(BaseSettings)` config.py L55](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/config.py#L55), [`settings = Settings()` L728](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/config.py#L728).
- **Template** — `pydantic_settings.BaseSettings` com `@computed_field` (deriva a DSN) e `@model_validator` (guarda de segredo default). [`class Settings(BaseSettings)` config.py L25](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/core/config.py#L25).
- **Prefect** — pydantic-settings profundamente nested + cadeia de prioridade de fontes própria + **profiles em TOML** (`~/.prefect/profiles.toml`). [`class PrefectBaseSettings(BaseSettings)` base.py L31](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/settings/base.py#L31), [`ProfilesCollection` profiles.py L114](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/settings/profiles.py#L114).

- **Warehouse** — dict de settings do Pyramid montado imperativamente de `os.environ` via helpers `maybe_set()` (~80 chamadas), sem schema/validação declarativa. - **Sentry** — **dois** sistemas: django settings estático de `os.environ` em `conf/server.py` + um registry de **options em runtime persistido em DB** (`register`/`get`/`set`), mutável sem deploy. **Anedota:** o registry de options em runtime (Sentry) — segunda fonte de verdade mutável, dirigida por escala; profiles TOML (Prefect); decryptors de Secret plugáveis (Dispatch). **Conflito com doutrina:** o options em DB do Sentry contradiz "config é env tipado imutável 12-factor" — mas é tradeoff de escala, anotado.

### 7. Async

**Sinal:** async **não** é garantido pra um backend de produção "FastAPI-shaped". O split é parelho: Polar + Prefect são async end-to-end; **Dispatch e o template são síncronos — os dois são FastAPI**; Warehouse e Sentry são síncronos (não-FastAPI). O próprio template do autor do FastAPI e o app FastAPI da Netflix rodam handlers `def` no threadpool. Async é escolhido quando há driver async ponta a ponta (asyncpg no Polar); senão `def` + threadpool é aceito em produção.

- **Polar** — async total: `AsyncSession`/`AsyncReadSession` como `NewType` distintos (separação read/write imposta pelo type-checker), asyncpg, endpoints/serviços/worker Dramatiq todos `async def`. [`AsyncReadSession = NewType(...)` postgres.py L16](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/kit/db/postgres.py#L16), [`@actor(...) async def order_created(...)` tasks.py L62](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/tasks.py#L62).
- **Prefect** — async ponta a ponta; única fuga síncrona é o Alembic, embrulhado em worker thread. [`await run_sync_in_worker_thread(alembic_upgrade)` interface.py L113](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/database/interface.py#L113).
- **Dispatch** — rotas **`def` síncronas** + SQLAlchemy síncrono; async só na casca ASGI/middleware. [`def get_incidents(...)` (sync) views.py L77](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/incident/views.py#L77).
- **Template** — totalmente síncrono (`create_engine` sync, `Session` sync, rotas `def`). [`def read_items(...)` items.py L14](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/api/routes/items.py#L14).
- **Warehouse** — WSGI puro, zero `async def`; background é Celery. [`application = configure().make_wsgi_app()` wsgi.py L3](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/wsgi.py#L3).
- **Sentry** — WSGI síncrono (Django); background **não é mais Celery** — é o sistema próprio `taskbroker-client`/taskworker. [`from taskbroker_client.task import P, R, Task` tasks/base.py L9](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/tasks/base.py#L9).

**Anedota:** a separação `AsyncSession`/`AsyncReadSession` por `NewType` imposta por tipo (Polar); Prefect embrulhando Alembic. **Conflito com doutrina / sabedoria recebida:** "FastAPI ⇒ async" é mito — o template (do autor do framework) e o Dispatch são síncronos. E "Sentry usa Celery" está desatualizado: no SHA fixado o `pyproject.toml` não tem Celery ([`taskbroker-client>=0.20.12` pyproject.toml L110](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/pyproject.toml#L110)).

### 8. Transações

**Sinal:** sessão-por-request é universal. **Quem commita** é que se divide: (a) commit-na-borda / unit-of-work (Warehouse `pyramid_tm`; Polar na dependency; Dispatch no middleware; Prefect no caller) vs. (b) commit-inline-na-operação (template, e o serviço do Dispatch também, Sentry `atomic` por write). Os apps mais limpos empurram o commit pra borda da request e deixam o serviço só `flush()`; os mais simples/antigos commitam inline. Polar é o mais limpo (serviços nunca commitam).

- **Polar** — sessão-por-request por middleware ASGI; a **dependency** `get_db_session` é a fronteira transacional — **ela** commita no sucesso / rollback na exceção; serviços só `flush()`. Também: `SELECT ... FOR UPDATE NOWAIT` → erro de domínio, e `UPDATE ... WHERE status='draft'` + rowcount como claim otimista. [`await session.commit()` (na dependency) postgres.py L107](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/postgres.py#L107), [claim otimista por rowcount repository.py L384](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/repository.py#L384).
- **Warehouse** — `zope.sqlalchemy` + pacote `transaction` + `pyramid_tm` (commit/abort no fim da request) + `pyramid_retry` (retry automático de `IntegrityError`). Unit-of-work-por-request de manual. [`pyramid_retry.mark_error_retryable(IntegrityError)` db.py L51](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/db.py#L51), [`config.include("pyramid_tm")` config.py L783](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/config.py#L770).
- **Prefect** — o **caller** (rota) decide a fronteira via `session_context(begin_transaction=True|False)`; models normalmente não commitam. [`async def session_context(self, begin_transaction=False, ...)` interface.py L149](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/database/interface.py#L149).
- **Dispatch** — `db_session_middleware` commita no sucesso na borda **e** faz rollback na exceção; **mas** o serviço também commita inline (propriedade dupla). Multi-tenant por schema translate. [`request.state.db.commit()` (no middleware) main.py L156](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/main.py#L156), [`db_session.commit()` (no serviço) service.py L210](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/incident/service.py#L210).
- **Sentry** — `transaction.atomic(router.db_for_write(Model))` explícito por write; writes cross-silo embrulhados em `outbox_context(...)` (padrão outbox: `CellOutbox`/`ControlOutbox`). [`with transaction.atomic(router.db_for_write(...))` organization_details.py L1224](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/core/endpoints/organization_details.py#L1224), [`with outbox_context(transaction.atomic(...))` organizationmember.py L254](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/models/organizationmember.py#L254).
- **Template** — sessão-por-request por context-manager `get_db`; commit **inline** no crud/rota. [`session.commit()` na rota items.py L70](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/api/routes/items.py#L70).

**Anedota:** `pyramid_tm`+`pyramid_retry` auto-retry de `IntegrityError` (Warehouse); outbox cross-silo (Sentry); `FOR UPDATE NOWAIT`→erro-de-domínio e claim otimista por rowcount (Polar); schema-por-tenant (Dispatch). **Conflito com doutrina:** Dispatch commita **em ambos** middleware e serviço (dono duplo da transação); a regra "caller é dono da transação" do Prefect tem violação documentada em campo (uma função de model commita sozinha, [models/flow_runs.py L761](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/models/flow_runs.py#L761)). "Um dono único de transação (uow)" é ideal só parcialmente sustentado.

### 9. Testes

**Sinal:** **testar contra um banco de verdade é universal** — ninguém mocka o DB nem testa o caminho principal só com fakes de repositório em memória. O isolamento é rollback de transação por teste (savepoint/nested no Dispatch, Warehouse; transação no Polar) **ou** delete de tabela (Prefect, template) — rollback é a maioria mais limpa. `factory_boy` é comum (Dispatch, Warehouse) mas não universal (Polar/Sentry fazem factory à mão; template nenhuma). Externos são fakeados por monkeypatch do import concreto, ou trocando o adapter do porto, ou trocando settings.

- **Dispatch** — Postgres real; isolamento por `begin_nested()`+`rollback()` por teste; ~60 factories `factory_boy` (`SQLAlchemyModelFactory` ligada a uma `Session` scoped); **portos fakeados registrando `Test*Plugin` no mesmo registry de plugin** — o seam de porto é exatamente onde o teste substitui. [`session.begin_nested(); yield; session.rollback()` conftest.py L133](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/tests/conftest.py#L133), [`class BaseFactory(SQLAlchemyModelFactory)` factories.py L76](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/tests/factories.py#L76), [`storage_plugin` registra `TestStoragePlugin` conftest.py L231](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/tests/conftest.py#L231).
- **Warehouse** — Postgres real (`pytest-postgresql` `DatabaseJanitor`), sessão presa a transação externa com **savepoint** (`join_transaction_mode="create_savepoint"`) sempre revertida; `factory_boy`; **os serviços são testados como integração** contra DB real, não com fakes dos portos. [`Session(bind=conn, join_transaction_mode="create_savepoint")` conftest.py L429](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/tests/conftest.py#L429), [`class WarehouseFactory(SQLAlchemyModelFactory)` common/db/base.py L8](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/tests/common/db/base.py#L8).
- **Polar** — Postgres real por worker xdist (migrado com Alembic de verdade uma vez); rollback de transação por teste; factories como **funções à mão** (não factory_boy); externos (Stripe) fakeados por `mocker.patch` no singleton de módulo — **sem fake-atrás-de-porto**, o alvo do mock é o próprio nome importado. [`transaction.rollback()` por teste fixtures/database.py L64](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/tests/fixtures/database.py#L64), [`mocker.patch("polar.order.service.stripe_service", new=mock)` test_service.py L162](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/tests/order/test_service.py#L162).
- **Prefect** — DB real (SQLite ou Postgres pelo scheme da URL), DB descartável por worker; isolamento por **delete de todas as tabelas** (`clear_db` autouse), não rollback; settings como seam de teste (`temporary_settings` troca messaging por in-memory). [`async def clear_db(...)` (delete autouse) fixtures/database.py L92](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/tests/fixtures/database.py#L92).
- **Sentry** — pytest + Django `TestCase` (DB real por teste) + mixin `Factories` **feito à mão** (com TODO no código de migrar pra factory_boy) sob uma hierarquia profunda de base-class em `testutils/cases.py`. [`# TODO(dcramer): consider moving to something more scalable like factoryboy` factories.py L402](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/testutils/factories.py#L402).
- **Template** — Postgres real, sessão **compartilhada** scope-session, limpeza por `DELETE` no fim — **sem isolamento por teste**; sem factories, dados montados por utils. [`db` fixture scope="session" + `delete` no teardown conftest.py L15](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/tests/conftest.py#L15).

**Anedota:** o porto-como-seam-de-teste do Dispatch (registrar um `Test*Plugin`) é a história de fake mais limpa; settings-como-seam do Prefect; **template sem isolamento por teste** (fraqueza real no template do próprio autor do framework). **Conflito com doutrina:** a doutrina do LUC de "use-case testado com fakes dos ports, sem DB" ([CONTEXT.md/ADR-0003](https://github.com/ThiagoPanini/life-under-control/blob/main/docs/adr/0003-nucleo-dominio-multi-borda.md)) é **contradita** — o Python de produção testa serviços/use-cases predominantemente como **integração contra Postgres real**; o subagente de Warehouse e o de Polar flagaram isso explicitamente. Teste unitário do serviço com fakes dos portos e sem DB é raro nesses codebases. Sinal forte, anotado sem resolver.

### 10. Topologia

**Sinal:** **pacote-por-feature/concept-de-domínio é a topologia dominante** (Dispatch, Polar, Warehouse, Sentry) — um pacote por conceito de negócio, cada um com suas próprias rotas/models/service/schemas, e uma convenção repetida de arquivos-por-papel dentro do concept. Um kernel compartilhado, livre de domínio, é comum (kit/ do Polar, common/ do Dispatch). Prefect é o outlier layer-first.

- **Polar** — pacote por concept (~80: account, benefit, checkout, order, subscription, webhook...) com convenção `__init__/auth/endpoints/repository/schemas/service/sorting/tasks`; kernel `kit/` livre de domínio. [`order/service.py` L1](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/order/service.py#L1), [`kit/repository/base.py` L1](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/kit/repository/base.py#L1).
- **Warehouse** — pacote por concept (accounts, packaging, organizations, oidc, macaroons...) com o mesmo "pentágono": `__init__/forms/interfaces/models/services/tasks/views`. [`warehouse/accounts/services.py` L79](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/services.py#L79).
- **Dispatch** — pacote por concept (incident/, case/, feedback/...), cada um com `models.py`/`service.py`/`flows.py`/`views.py` + role modules; sub-concepts nested (incident/priority/, incident/severity/). [`src/dispatch/incident/flows.py` L389](https://github.com/Netflix/dispatch/blob/dd2837e82a0bf5565b1b4b4b91ea30b7262d4061/src/dispatch/incident/flows.py#L389).
- **Sentry** — ~140 pacotes por feature; os maduros partem internamente em `models/`/`api/`/`services/`; refactor em curso centralizando endpoints num `core/endpoints/`. [`src/sentry/replays/usecases/delete.py` L70](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/replays/usecases/delete.py#L70).
- **Prefect** — **layer-first** no topo (`api/`, `models/`, `schemas/`, `services/`, `orchestration/`, `database/`) com **nome de arquivo por feature dentro de cada camada** (`flow_runs.py` repetido em cada layer) — o inverso de todos os outros. [`schemas/core.py` L1](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/schemas/core.py#L1).
- **Template** — plano por camada (`api/routes/`, `core/`, `crud.py`, `models.py`) — mínimo, por ser pequeno. [`backend/app/models.py` L14](https://github.com/fastapi/full-stack-fastapi-template/blob/4d3d5e92c1ea6b3fa0fab02c41124844ec45bca8/backend/app/models.py#L14).

**Anedota:** Prefect layer-first com nome-de-arquivo-por-feature (contra-exemplo que escala bem); a re-centralização `core/` em curso no Sentry; Dispatch aninhando sub-concepts. **Confirmação com doutrina:** este eixo **confirma** o instinto "pacote por concept, borda fina por concept, kernel `shared/` mínimo" do `apps/api` ([ADR-0003](https://github.com/ThiagoPanini/life-under-control/blob/main/docs/adr/0003-nucleo-dominio-multi-borda.md)); Prefect prova que layer-first também escala.

## Conflitos com doutrina (agregado, não resolvido)

A adjudicação destes é de outros tickets/grillings. Aqui só se anota onde a prática de produção contradiz doutrina conhecida (Clean/Hexagonal/DDD) e/ou a doutrina atual do `apps/api`.

1. **Portos não são universais** — existem só onde há uma 2ª implementação real (multi-backend, integração plugável, RPC cross-silo). Contradiz "porta pra toda dependência". Polar até tem um `Protocol` de repositório que **não** usa como seam ([base.py L35](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/kit/repository/base.py#L35)).
2. **Domínio rico ausente** — regra de negócio é função procedural sobre a sessão, às vezes **no próprio modelo ORM** (Polar [order.py L438](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/models/order.py#L438); `hybrid_property` de política em [organization.py L714](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/models/organization.py#L714)) ou no Manager Django (Sentry [project.py L176](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/models/project.py#L176)). DDD tático (agregado com comportamento) não aparece.
3. **DTO acoplado ao ORM de propósito** — `from_attributes`/SQLModel/`serialize()` colam o DTO ao shape do ORM; Prefect docstring o schema de API como "ORM representation" ([bases.py L216](https://github.com/PrefectHQ/prefect/blob/60025750fb7ea1d69fef8a1c8b1e013cdc8149ae/src/prefect/server/utilities/schemas/bases.py#L216)). Contradiz "domínio ≠ persistência ≠ DTO + mapeamento".
4. **Erro de domínio carrega HTTP** — `PolarError.status_code` ([exceptions.py L11](https://github.com/polarsource/polar/blob/034c7474e763201b934369756fc221ca7e9848d2/server/polar/exceptions.py#L11)), hierarquia do Sentry sobre `APIException`, status na docstring do Prefect. Contradiz "erro de domínio agnóstico de transporte". Nenhum usa problem+json (contraste com [#197](https://github.com/ThiagoPanini/life-under-control/issues/197)).
5. **Commit dentro do use-case / dono duplo de transação** — template e serviço do Dispatch commitam inline; Dispatch commita em middleware **e** serviço; violação em campo no "caller-owns" do Prefect. Contradiz unit-of-work único na borda.
6. **Use-case testado como integração com Postgres real, não com fakes dos portos** — universal nos seis; Warehouse/Polar flagados explicitamente. Contradiz a doutrina de teste de use-case com fakes sem DB ([ADR-0003](https://github.com/ThiagoPanini/life-under-control/blob/main/docs/adr/0003-nucleo-dominio-multi-borda.md)).
7. **Validação com efeito colateral** — `LoginForm.validate_password` do Warehouse dispara e-mail e desabilita senha ([forms.py L454](https://github.com/pypi/warehouse/blob/f576313a510043e07f72dd4db75c4084025566da/warehouse/accounts/forms.py#L454)). Contradiz "validação é etapa pura".
8. **DI = singleton no import / service-locator, não construtor/container** (exceto Warehouse). Contradiz "injete tudo por construtor".
9. **Config mutável em runtime** — options em DB do Sentry ([defaults.py L26](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/src/sentry/options/defaults.py#L26)). Contradiz "config é env tipado imutável".
10. **Sabedoria recebida desatualizada** — "FastAPI ⇒ async" é falso (template e Dispatch são síncronos); "Sentry usa Celery" é falso no SHA fixado (é `taskbroker-client`, [pyproject.toml L110](https://github.com/getsentry/sentry/blob/c4a0d0de19ef2e0132c02ae3b5745520861b43d2/pyproject.toml#L110)).

## Síntese: sinais fortes vs. anedotas

Sinais fortes (recorrentes entre codebases independentes, framework-agnósticos):

- Use-case procedural que **recebe a sessão** (função livre ou classe stateless), sem agregado de domínio rico — todos os seis.
- Schemas separados de entrada/saída por concept, com o DTO de saída construído **direto do ORM** — todos exceto o WTForms de Warehouse.
- Hierarquia de erro própria + **tradução central pra HTTP na borda** — quatro dos cinco apps (o template levanta HTTPException inline).
- Sessão-por-request; **quem commita** divide os codebases (uow-na-borda vs. inline).
- **Teste contra DB real** é universal; fake-atrás-de-porto sem DB é raro.
- **Pacote-por-feature/concept** é a topologia dominante (Prefect é o contra-exemplo layer-first).
- DI de dois níveis: `Depends`/`find_service` do framework + singletons de módulo; container dedicado é raríssimo.

Anedotas (idiossincrasia de um só codebase):

- zope.interface + container IoC de verdade (só Warehouse).
- Erro de domínio embutido na reason-phrase HTTP (só Warehouse, API legada).
- Options mutável em DB em runtime (só Sentry) e taskbroker próprio no lugar de Celery (só Sentry).
- Separação read/write de `AsyncSession` por `NewType` imposta por tipo, e claim otimista por rowcount (só Polar).
- Portos por `abc.ABC` de backend de DB + profiles TOML (só Prefect).
- Orquestração de efeitos externos em `flows.py` com `@background_task` disparado por `BackgroundTasks`, e portos como base-class + registry por metaclasse (só Dispatch).

## Limites do método

Cada codebase foi lido num único commit fixado; práticas mudam no tempo (o Sentry, em particular, está em refactor visível — `core/endpoints/`, taskbroker no lugar de Celery). "Sinal" aqui é recorrência entre estes seis espécimes, não prova de universalidade estatística. O recorte é deliberadamente enviesado pra backends HTTP de domínio "FastAPI-shaped ou equivalentes"; não fala de data-pipelines, ML-serving ou libs. Nenhum achado prescreve o que o LUC deve fazer — o radar descreve o campo; a decisão é dos tickets de adjudicação do mapa [#219](https://github.com/ThiagoPanini/life-under-control/issues/219).
